# BMAD Method — Step-by-Step Workflow Walkthrough

> A practical guide for engineering teams: how to initiate each phase, what's expected of you, what the agent produces, and where artifacts are saved.

---

## How BMAD Works in Practice

Each BMAD skill is invoked as a **slash command** in Claude Code (e.g., `/bmad-create-prd`). When you invoke a skill:

1. An **AI agent persona** activates (e.g., "John the Product Manager")
2. The agent **reads prior artifacts** (product brief, PRD, architecture, etc.) automatically
3. The agent **guides you** through a structured conversation or runs autonomously
4. The agent **produces an output artifact** (a markdown file) saved to the project's output folder
5. That artifact becomes **input for the next phase**

**Important:** BMAD recommends running each skill in a **fresh Claude Code session** to keep context clean. Start a new session, navigate to your project directory, and invoke the skill.

**Output locations:**
- Planning artifacts: `{output_folder}/planning-artifacts/`
- Implementation artifacts: `{output_folder}/implementation-artifacts/`
- Test artifacts: `{output_folder}/test-artifacts/`

## Phase 1: Analysis

### Step 1 — Create Product Brief

| | Details |
|--|---------|
| **Command** | `/bmad-product-brief` |
| **Agent** | Mary (Business Analyst) |
| **Purpose** | Capture your product idea as a 1-2 page executive brief |
| **Prerequisites** | None — this is where it all starts |
| **Output** | `{planning_artifacts}/product-brief.md` |

#### What You Need to Provide

- **Your product/project idea** — describe it in your own words, brain dump everything
- **Any existing documents** — research, notes, prior specs (optional, agent will scan them)
- **Answers to discovery questions** — the agent asks about vision, users, market, differentiation

#### Modes

| Mode | How to Invoke | What Happens |
|------|---------------|--------------|
| **Guided** (default) | `/bmad-product-brief` | Conversational — agent walks you through 5 stages with questions |
| **Autonomous** | `/bmad-product-brief --autonomous` | Agent produces the brief with minimal interaction |
| **Yolo** | `/bmad-product-brief --yolo` | Agent drafts the entire brief first, then walks you through refinements |

#### The 5 Stages

1. **Understand Intent** — Agent asks: what are we building and why?
2. **Contextual Discovery** — Agent analyzes your docs and does web research (runs in parallel)
3. **Guided Elicitation** — Agent fills gaps by asking smart questions about vision, users, market, success
4. **Draft & Review** — Agent drafts the brief, runs it through skeptic + opportunity reviewers
5. **Finalize** — Agent polishes and saves. Optionally creates a "distillate" (compressed version for downstream AI consumption)

#### Tips

- Don't hold back during the brain dump — agent captures everything, even out-of-scope details
- At every pause, agent asks "Anything else?" — this consistently draws out extra context
- The distillate is valuable for large projects — it preserves all overflow detail for the PRD phase

## Phase 2: Planning

### Step 2 — Create PRD (Product Requirements Document)

| | Details |
|--|---------|
| **Command** | `/bmad-create-prd` |
| **Agent** | John (Product Manager) |
| **Purpose** | Turn the product brief into a full, structured requirements document |
| **Prerequisites** | Product brief (from Step 1) |
| **Output** | `{planning_artifacts}/prd.md` |

#### What You Need to Provide

- **Respond to menus and guided prompts** — the agent walks you step by step
- **Domain expertise** — you know your business; the agent knows the structure
- **Approval at each step** — press `C` to continue to the next step

#### The 12 Steps

| Step | What Happens | What You Do |
|------|-------------|-------------|
| 1. Init | Agent discovers input docs, creates document skeleton | Confirm inputs |
| 2. Discovery | Agent classifies project type and domain (fintech, web app, etc.) | Confirm classification |
| 3. Vision | Agent facilitates product vision discovery | Share your vision, answer prompts |
| 4. Executive Summary | Agent generates executive summary | Review and approve |
| 5. Success Criteria | Agent defines user/business/technical success metrics | Validate metrics are right |
| 6. User Journeys | Agent maps user journeys as story-based narratives | Confirm journeys cover your scenarios |
| 7. Domain Requirements | Agent adds domain-specific requirements (e.g., fintech compliance) | Validate completeness |
| 8. Innovation | Agent identifies competitive differentiators (optional) | Confirm or skip |
| 9. Project Type | Agent asks project-type-specific questions (API, web app, etc.) | Answer questions |
| 10. Scoping | Agent facilitates MVP scoping exercise | Define in/out boundaries |
| 11. Functional & Non-Functional Requirements | Agent synthesizes FRs and NFRs from everything above | Review for completeness |
| 12. Polish | Agent polishes document flow, deduplicates, checks coherence | Final approval |

#### BMAD PRD Quality Standards

The PRD follows strict BMAD quality rules:
- **High information density** — every sentence carries weight, zero filler
- **SMART requirements** — Specific, Measurable, Attainable, Relevant, Traceable
- **Full traceability** — Vision → Success Criteria → User Journeys → FRs
- **Zero anti-patterns** — no "user-friendly", no "fast", no implementation leakage (no tech names in requirements)
- **Dual-audience** — readable by humans AND optimized for downstream AI agents

### Step 3 — Validate PRD (Quality Gate)

| | Details |
|--|---------|
| **Command** | `/bmad-validate-prd` |
| **Agent** | Validation Architect |
| **Purpose** | Quality gate — validate the PRD meets BMAD standards before proceeding |
| **Prerequisites** | Completed PRD |
| **Output** | Validation report with findings |

#### What You Need to Provide

- **The PRD path** (agent auto-discovers it if in the standard location)
- **Responses to findings** — fix issues the validator surfaces

#### What Gets Validated (13 checks)

1. Document discovery and format detection
2. BMAD Standard format compliance
3. Information density (anti-pattern scanning)
4. Product Brief coverage (did the PRD address everything in the brief?)
5. FR/NFR measurability (are all requirements testable?)
6. Traceability chain (does everything link up?)
7. Implementation leakage (no tech names in requirements)
8. Domain compliance (fintech/healthcare/etc. requirements present?)
9. Project-type compliance (web app/API/mobile requirements covered?)
10. SMART scoring
11. Holistic quality assessment
12. Completeness validation (final gate)
13. Report finalization

## Phase 3: Solutioning

### Step 5 — Create Architecture

| | Details |
|--|---------|
| **Command** | `/bmad-create-architecture` |
| **Agent** | Winston (Architect) |
| **Purpose** | Document technical architecture and design decisions |
| **Prerequisites** | PRD, optionally UX design |
| **Output** | Architecture document in `{planning_artifacts}/` |

#### What You Need to Provide

- **Domain expertise and product context** — agent reads the PRD but needs your input on constraints
- **Approve architectural decisions** — agent proposes, you validate
- This is a **peer collaboration** — agent facilitates, you decide

#### Key Areas Covered

- System architecture (components, services, data flow)
- Technology stack decisions
- Integration architecture (how it connects to existing systems)
- Security architecture
- Infrastructure and deployment
- Architectural Decision Records (ADRs)

| | Details |
|--|---------|
| **Command** | `/bmad-create-epics-and-stories` |
| **Agent** | Bob (Scrum Master) |
| **Purpose** | Break the PRD and architecture into implementable epics and user stories |
| **Prerequisites** | PRD + Architecture document |
| **Output** | Epics and stories document in `{planning_artifacts}/` |

#### What You Need to Provide

- **Product vision and business priorities** — agent needs your input on sequencing
- **Respond to menus** — step-file guided workflow

#### What Gets Produced

- Epics (large feature groups)
- User stories within each epic
- BDD acceptance criteria (Given/When/Then) per story
- Priority and dependency ordering

### Step 7 — Check Implementation Readiness (Quality Gate)

| | Details |
|--|---------|
| **Command** | `/bmad-check-implementation-readiness` |
| **Agent** | Expert PM + Scrum Master (traceability specialist) |
| **Purpose** | Final quality gate before coding — are PRD, Architecture, and Epics/Stories aligned? |
| **Prerequisites** | PRD + Architecture + Epics/Stories (all three must exist) |
| **Output** | Readiness validation report |

#### What You Need to Provide

- **Respond to validation findings** — fix gaps before proceeding
- This is a **gate** — do not skip this step

#### What Gets Checked

- Do epics/stories account for ALL PRD requirements?
- Is the architecture sufficient for all requirements?
- Are stories implementable given the architecture?
- Are there gaps, conflicts, or missing coverage?

## Phase 4: Implementation

### Step 8 — Sprint Planning

| | Details |
|--|---------|
| **Command** | `/bmad-sprint-planning` |
| **Agent** | Bob (Scrum Master) |
| **Purpose** | Generate the sprint tracking file from epics |
| **Prerequisites** | Epics document in `{planning_artifacts}/` |
| **Output** | `{implementation_artifacts}/sprint-status.yaml` |

#### What You Need to Provide

- **Nothing** — this runs fully autonomously
- Agent parses epics, extracts all work items, and builds the tracking structure

#### Status Flow

```
Epic:    backlog → in-progress → done
Story:   backlog → ready-for-dev → in-progress → review → done
Retro:   optional → done
```

---

### Step 9 — Create Story (Prepare for Development)

| | Details |
|--|---------|
| **Command** | `/bmad-create-story` or `/bmad-create-story 2-4` (specific story) |
| **Agent** | Story Context Engine |
| **Purpose** | Create a comprehensive story file with all context a developer agent needs |
| **Prerequisites** | sprint-status.yaml, epics, PRD, architecture, UX design |
| **Output** | `{implementation_artifacts}/{story-key}.md` (e.g., `1-2-user-authentication.md`) |

#### What You Need to Provide

- **Optionally: specific story** — e.g., `/bmad-create-story 1-3` for epic 1, story 3
- **If no story specified**, agent auto-discovers the first `backlog` story from sprint-status.yaml
- **Minimal interaction** — runs near-autonomously, only asks if docs are missing

#### What the Agent Does

1. Loads ALL core artifacts exhaustively (PRD, architecture, UX, epics)
2. Analyzes architecture for developer guardrails
3. Does web research for latest technical specifics
4. Creates a comprehensive story file from template
5. Updates sprint-status.yaml (story → `ready-for-dev`)

---

### Step 10 — Dev Story (Implement)

| | Details |
|--|---------|
| **Command** | `/bmad-dev-story` or `/bmad-dev-story path/to/story.md` |
| **Agent** | Amelia (Developer) |
| **Purpose** | Implement the story — write code, tests, and validate |
| **Prerequisites** | Story file in `ready-for-dev` status |
| **Output** | Implemented code + updated story file + updated sprint-status.yaml |

#### What You Need to Provide

- **Optionally: story file path** — otherwise agent picks the first `ready-for-dev` story
- **Minimal interaction** — agent runs continuously until done or halted

#### How It Works

- Agent loads story file, project context, and all relevant artifacts
- Implements tasks using **red-green-refactor** cycle (write failing test → make it pass → refactor)
- Updates task checkboxes in the story file as it progresses
- Marks story as `review` when complete
- **HALT conditions** (agent stops and asks for help):
  - New dependencies discovered beyond the spec
  - 3 consecutive failures on a task
  - Missing configuration or access

#### Post-Code-Review Loop

If code review finds issues, `/bmad-dev-story` detects the review follow-ups and resumes with fixes.

---

### Step 11 — Code Review

| | Details |
|--|---------|
| **Command** | `/bmad-code-review` |
| **Agent** | Elite Code Reviewer |
| **Purpose** | Adversarial review of implemented code |
| **Prerequisites** | Story in `review` status, implemented code changes |
| **Output** | Structured review findings with triage categories |

#### What You Need to Provide

- **Respond at checkpoints** — agent halts for human input at key decision points

#### Review Layers (Run in Parallel)

| Reviewer | Focus |
|----------|-------|
| **Blind Hunter** | Bugs, logic errors, security issues |
| **Edge Case Hunter** | Boundary conditions, unhandled paths |
| **Acceptance Auditor** | Does the code meet the story's acceptance criteria? |

#### After Review

- If issues found → go back to `/bmad-dev-story` (it auto-detects review follow-ups)
- If approved → move to next story (`/bmad-create-story`) or retrospective if epic is complete

### Step 13 — Retrospective (Post-Epic Review)

| | Details |
|--|---------|
| **Command** | `/bmad-retrospective` |
| **Agent** | Bob (Scrum Master) facilitating a full simulated team |
| **Purpose** | Review completed epic — lessons learned, what went well, what didn't |
| **Prerequisites** | Completed epic (all stories `done` in sprint-status.yaml) |
| **Output** | `{implementation_artifacts}/epic-{N}-retro-*.md` |

#### What You Need to Provide

- **Confirm which epic** to review
- **Participate as "Project Lead"** in the team retrospective discussion
- Answer questions about what went well, what didn't, improvements for next epic

#### The Simulated Team

The retrospective runs as a **party mode** discussion with personas:

| Name | Role |
|------|------|
| Bob | Scrum Master (facilitator) |
| Alice | Product Owner |
| Charlie | Senior Developer |
| Dana | QA Engineer |
| Elena | Junior Developer |

All dialogue is formatted as `Name (Role): dialogue` — it feels like a real team retro.

## Quick Flow (Bypass Full Pipeline)

### Quick Dev — For Bug Fixes, Small Changes, Ad-Hoc Work

| | Details |
|--|---------|
| **Command** | `/bmad-quick-dev` |
| **Agent** | Developer |
| **Purpose** | Implement any intent without the full BMAD ceremony |
| **Prerequisites** | None required (uses project-context.md if available) |
| **Output** | `{implementation_artifacts}/spec-wip.md` + implemented code |

#### What You Need to Provide

- **Your intent** — "fix the login bug", "add rate limiting to the API", "refactor the auth module"
- Agent creates a lightweight spec, then implements

#### Scope Rules

- Targets a **single user-facing goal**
- If your request has multiple goals, agent splits them
- Spec is 900-1600 tokens — concise and actionable

## The Implementation Loop (Repeat Per Story)

Once you're in Phase 4, the cycle for each story is:

```
/bmad-create-story          → Prepares the story file
/bmad-dev-story             → Implements the code
/bmad-code-review           → Reviews the code
   ↓ (if issues found)
/bmad-dev-story             → Fixes issues (auto-detects review follow-ups)
/bmad-code-review           → Re-reviews
   ↓ (if approved)
/bmad-sprint-status         → Check progress, get next recommendation
/bmad-create-story          → Next story...
```

When all stories in an epic are `done`:

```
/bmad-retrospective         → Team retrospective for the epic
```
